EasyAccess — 开发指导文档

"Already open. One click away." / "已打开的文件夹，一键即达。" · v1.0 Draft · 2026-06-17

EasyAccess 是一款 Windows 常驻后台工具，在检测到文件打开/保存对话框时，在对话框底部自动显示后台已打开的资源管理器文件夹列表，用户一键即可导航到目标文件夹，消除反复手动浏览目录的重复操作。面向所有 Windows 用户，兼容 Win10 1903+ 和 Win11。

1. 整体架构设计

1.1 架构分层

EasyAccess 采用四层架构，单进程运行，职责清晰分离：

┌──────────────────────────────────────────────────────┐
│                    UI Layer                           │
│  OverlayWindow (Win32 宿主 + XAML Islands/WinUI 3)   │
│  TrayIcon (Shell_NotifyIcon)                          │
├──────────────────────────────────────────────────────┤
│                Core Logic Layer                       │
│  DialogDetector  ·  FolderCollector  ·  Navigator     │
│  WhitelistEngine                                      │
├──────────────────────────────────────────────────────┤
│                System Layer                           │
│  WinEventHook · UIAutomation · COM Shell · SendInput  │
├──────────────────────────────────────────────────────┤
│              Infrastructure Layer                     │
│  SingleInstance · AutoStart · ConfigManager · Logger  │
└──────────────────────────────────────────────────────┘

1.2 线程模型

线程职责消息循环
Main ThreadWin32 消息泵、SetWinEventHook 回调、TrayIcon、窗口位置同步定时器GetMessage / DispatchMessage
UI ThreadXAML Islands 渲染、Overlay 窗口绘制DispatcherQueue (WinUI 3)
Collector ThreadIShellWindows 遍历、Win11 多标签路径获取事件触发，非轮询
Input ThreadSendInput / UI Automation 导航注入操作按需创建

1.3 数据流

核心数据流：Windows Event → DialogDetector 检测 → FolderCollector 收集 → OverlayManager 显示 → Navigator 注入导航

[Windows Event]
    ↓ SetWinEventHook(EVENT_OBJECT_CREATE / EVENT_SYSTEM_FOREGROUND)
DialogDetector.detect(hwnd)
    ├── 标准对话框: GetClassNameW == "#32770" + 控件特征验证
    └── 白名单应用: WhitelistEngine → 应用专用检测逻辑
    ↓ 检测到文件对话框
FolderCollector.getOpenFolders()
    ├── IShellWindows → 遍历资源管理器窗口路径
    ├── Win11: COM 接口获取多标签路径
    └── SHChangeNotifyRegister → 增量更新
    ↓ 文件夹列表就绪
OverlayManager.show(hwndDialog, folders)
    ├── 创建/复用 overlay Win32 宿主窗口
    ├── SetWindowLongPtrW(GWL_HWNDPARENT, hwndDialog)
    ├── 计算位置: dialog 底部居中
    └── WinUI 3 ListView 渲染文件夹列表
    ↓ 用户点击文件夹
Navigator.navigateTo(targetPath)
    ├── UI Automation: 查找对话框编辑框
    ├── 设置路径值 (uia_set_value / clipboard_set)
    └── 确认导航 (Invoke 按钮 / SendInput Enter)
    ↓ 对话框关闭
    SetWinEventHook(EVENT_OBJECT_DESTROY) → OverlayManager.hide()

2. 技术栈选型

2.1

语言与编译器

C++20 + MSVC (Visual Studio 2022)。使用 C++/WinRT 投影访问 Windows Runtime API。

选择理由：常驻后台进程，无 GC pause；原生 Win32 API 零开销调用；编译产物体积小。

2.2

UI 渲染：Win32 宿主 + XAML Islands + WinUI 3

Overlay 窗口采用混合架构：

  • 外层：Win32 窗口（CreateWindowExW，WS_POPUP 样式），完全控制 WndProc、HWND 属性、窗口层级
  • 内层：通过 DesktopWindowXamlSource（XAML Islands）嵌入 WinUI 3 XAML 内容，使用 WinUI 3 原生 ListView + Fluent Design 渲染

这样既拥有 Win32 窗口的完全控制权（设置 GWL_HWNDPARENT、精确定位、消息拦截），又享受 WinUI 3 的原生 Win11 风格渲染（圆角、Acrylic 背景、流畅动画、暗色/亮色主题自动适配）。

3. 核心模块：文件对话框检测

这是整个系统最关键、最复杂的模块。采用两层检测策略：标准对话框检测 + 白名单应用检测。不在白名单中的非标准对话框不触发，确保零误判。

3.1 事件监听

通过 SetWinEventHook 监听两个事件：

  • EVENT_OBJECT_CREATE — 新窗口创建时触发，用于第一时间检测新打开的对话框
  • EVENT_SYSTEM_FOREGROUND — 前台窗口切换时触发，用于检测已存在但刚获得焦点的对话框

Hook 回调必须在 5ms 内返回，避免阻塞系统消息队列。耗时操作（UI Automation 树遍历）post 到工作线程异步执行。

3.2 Layer 1：标准对话框检测

约 70% 的应用使用 Windows 标准文件对话框（GetOpenFileName / IFileDialog）。

检测流程:
  1. GetClassNameW(hwnd) == L"#32770"          // 标准对话框类名
  2. 枚举子控件 (EnumChildWindows):
     - 存在 ComboBoxEx32 (文件类型下拉框)
     - 存在 Edit (文件名输入框)
     - 存在 Button 且文本含 "打开|Open|保存|Save|另存"
  3. 至少命中 2 个特征 → 确认为文件对话框

3.3 Layer 2：白名单应用检测

对于自绘文件对话框的应用（Chrome、Edge、VS Code、Office 等），使用应用专用的检测逻辑。白名单配置为外部 JSON 文件，无需重新编译。

每个白名单条目定义：

  • process_name：进程名匹配
  • window_class：窗口类名匹配（可选）
  • detect.uia_conditions：UI Automation 树中的控件特征组合
  • detect.min_match_count：最少命中数

检测逻辑：前台窗口进程名 → 匹配白名单条目 → 按该条目的 detect 规则验证 UI Automation 树 → 命中则确认为文件对话框。

3.4 提权窗口处理

UAC 安全边界：以管理员权限运行的应用（如管理员 PowerShell 中的文件对话框），非提权进程无法访问其 UI Automation 树。

处理方案：检测到目标对话框的进程 Token 为管理员权限且 EasyAccess 自身为非提权时，弹出 Toast 通知提示用户以管理员身份重启 EasyAccess。

4. 核心模块：文件夹列表收集

4.1 设计原则

  • 事件驱动，按需收集：对话框检测到时才触发一次 IShellWindows 遍历，不轮询
  • 增量更新：对话框存在期间，通过 SHChangeNotifyRegister 监听文件系统变化做增量更新
  • 即时释放：对话框关闭后立即停止监听，释放资源

4.2 IShellWindows 遍历

通过 COM 接口 IShellWindows（CLSID_ShellWindows）获取所有资源管理器窗口：

CoCreateInstance(CLSID_ShellWindows)
  → IShellWindows::get_Count()
  → 遍历每个窗口:
     → QueryInterface(IWebBrowser2)
     → get_LocationURL() → URL 解码为本地路径
     → 验证路径存在且为目录
     → 收集: { path, displayName, windowTitle }

4.3 Win11 多标签支持

Windows 11 资源管理器支持多标签页。每个标签页可能打开不同的文件夹。

获取多标签路径的方法：

  • 通过 IShellBrowser 接口从每个标签页获取当前浏览路径
  • 使用 ITabEventHandler（如果可用）监听标签切换事件
  • Fallback：通过 UI Automation 遍历 Tab 控件子项，获取每个标签的路径文本

Win10 不支持原生多标签，跳过此步骤。

4.4 增量更新策略

对话框存在期间注册 SHChangeNotifyRegister，监听以下事件：

  • SHCNE_MKDIR — 新文件夹创建
  • SHCNE_RENAMEFOLDER — 文件夹重命名
  • SHCNE_RMDIR — 文件夹删除

收到事件后更新 FolderCollector 内部缓存，Overlay 通过 DispatcherQueue 同步刷新 UI。

5. 核心模块：UI 覆盖层

5.1 窗口创建与层级控制

Overlay 是一个 Win32 宿主窗口，内部通过 XAML Islands 嵌入 WinUI 3 内容：

1. CreateWindowExW(WS_EX_TOOLWINDOW | WS_EX_TOPMOST,
                   className, L"EasyAccess",
                   WS_POPUP, ...)
   // WS_EX_TOOLWINDOW: 不在任务栏显示
   // WS_EX_TOPMOST: 确保在对话框上方
   // WS_POPUP: 弹出窗口，无标题栏

2. SetWindowLongPtrW(hwndOverlay, GWL_HWNDPARENT, (LONG_PTR)hwndDialog)
   // 设置为对话框的 owned window
   // z-order 自动跟随：overlay 始终在 owner 之上
   // 对话框关闭时 overlay 自动处理

3. DesktopWindowXamlSource xamlSource;
   xamlSource.Initialize(hwndOverlay);
   // 在 Win32 窗口内初始化 XAML 渲染引擎

4. xamlSource.Content(CreateOverlayContent());
   // 设置 WinUI 3 XAML 内容

5.2 定位与跟随

Overlay 定位在对话框底部，宽度与对话框一致（或略窄，两侧各留 16px 边距）。

跟随机制：使用 SetWinEventHook(EVENT_OBJECT_LOCATIONCHANGE) 监听对话框位置变化，在回调中用 SetWindowPos 同步更新 Overlay 位置。

GWL_HWNDPARENT 的优势：设置了 owned window 关系后，Overlay 的 z-order 自动跟随 owner（对话框）。当对话框被其他窗口遮挡时，Overlay 也一起被遮挡，不会出现显示撕裂。移动对话框时，Windows 会自动移动 owned window，EVENT_OBJECT_LOCATIONCHANGE 回调用于精确定位修正。

5.3 WinUI 3 XAML 内容

Overlay 内容使用 WinUI 3 原生控件，确保与 Win11 系统 UI 完全一致：

  • ListView / ItemsRepeater — 文件夹列表，支持虚拟化
  • AcrylicBackdrop — 半透明模糊背景，系统级材质
  • 圆角 + 阴影 — WinUI 3 默认 CornerRadius 和 DropShadow
  • 主题适配 — 通过 Application.RequestedTheme 跟随系统暗色/亮色

5.4 列表项设计

  • 每行显示：文件夹名称（粗体） + 完整路径（灰色小字，第二行）
  • Hover 高亮：WinUI 3 原生 ListViewItem PointerOver 样式
  • 点击反馈：Pressed 样式 + 导航执行
  • 显示数量：最多 10 项可见，超出部分滚动
  • 高度自适应：≤ 3 个文件夹时自适应内容高度，> 3 个固定最大高度 320px
  • 路径过长时中间截断显示：C:\Users\...\Documents

5.5 交互行为

  • 点击 Overlay 外部（对话框内区域）→ Overlay 不消失，只有对话框关闭才消失
  • Overlay 不抢焦点（WS_EX_NOACTIVATE），不影响对话框内正常操作
  • 键盘事件透传到对话框（如直接在对话框中输入文件名）

5.6 DPI 感知

EasyAccess 声明为 Per-Monitor DPI Aware V2。Overlay 创建时获取对话框所在显示器的 DPI，按比例缩放尺寸和字体。对话框跨显示器移动时，重新获取 DPI 并调整。

6. 核心模块：导航注入

6.1 基本流程

用户点击 Overlay 中的文件夹后，EasyAccess 需要在目标对话框中"输入"路径并确认导航。

6.2 导航策略

每个白名单应用在 JSON 中定义自己的 navigate 策略。主流两种方式：

策略 A：UI Automation 直接设值

  • 通过 UI Automation 找到对话框中的文件名 Edit 控件
  • ValuePattern.SetValue(targetPath)
  • 找到"打开/保存"按钮 → InvokePattern.Invoke()

适用：标准对话框、大部分使用 UI Automation 的应用

优点：精准、可靠、无副作用

策略 B：剪贴板 + SendKeys

  • 保存当前剪贴板内容
  • 将目标路径写入剪贴板
  • 聚焦 Edit 控件 → Ctrl+V 粘贴
  • SendInput 模拟 Enter 键
  • 恢复原剪贴板内容

适用：对 UI Automation 响应不完整的应用

优点：兼容性广

6.3 标准对话框的导航

标准对话框（#32770）使用策略 A：

1. FindWindowExW(hwndDialog, NULL, L"Edit", NULL)  // 文件名编辑框
2. SendMessage(hwndEdit, WM_SETTEXT, 0, (LPARAM)targetPath)
3. FindWindowExW(hwndDialog, NULL, L"Button", NULL) // 遍历找到"打开"按钮
4. SendMessage(hwndButton, BM_CLICK, 0, 0)

6.4 UIA 导航的可靠性保证

注意：不同应用的 UI Automation 实现质量参差不齐。白名单中每个应用的 navigate 策略需要实际测试验证。建议在白名单中增加 verified 字段标记已测试的配置，未验证的配置在 UI 上标注"实验性支持"。

7. 基础设施层

7.1 防多开（SingleInstance）

使用 Named Mutex 实现：

HANDLE hMutex = CreateMutexW(NULL, TRUE, L"Global\\EasyAccess_SingleInstance");
if (GetLastError() == ERROR_ALREADY_EXISTS) {
    // 已有实例运行，激活已有窗口或直接退出
    HWND existing = FindWindowW(L"EasyAccess_Hidden", NULL);
    if (existing) SendMessage(existing, WM_USER + 1, 0, 0);
    return 0;
}

7.2 开机自启（AutoStart）

使用 HKCU Registry Run Key，不需要管理员权限：

注册表路径: HKEY_CURRENT_USER\Software\Microsoft\Windows\CurrentVersion\Run
键名: EasyAccess
键值: "C:\path\to\EasyAccess.exe" --silent

实现:
  RegOpenKeyExW(HKEY_CURRENT_USER, RunPath, 0, KEY_SET_VALUE, &hKey)
  RegSetValueExW(hKey, L"EasyAccess", 0, REG_SZ, exePath, ...)
  // --silent 参数: 启动时不显示任何 UI，直接进入后台
  // 用户通过托盘菜单启用/禁用自启

7.3 配置管理（ConfigManager）

JSON 格式配置文件，存储在 exe 同目录。如果 exe 在只读目录（如 Program Files），退化到 %APPDATA%\EasyAccess\。

config.json:
{
  "autostart": true,
  "show_overlay_on_detect": true,
  "max_overlay_items": 10,
  "overlay_position": "bottom",
  "theme": "system",
  "log_level": "warn",
  "whitelist_file": "whitelist.json"
}

7.4 日志（Logger）

轻量文件日志，支持级别控制：

  • error：致命错误、COM 初始化失败
  • warn：检测到对话框但白名单匹配失败、UAC 权限不足
  • info：对话框检测/关闭、文件夹列表更新
  • debug：每个窗口事件、UI Automation 树遍历结果、导航注入步骤

日志文件：logs/EasyAccess_YYYYMMDD.log，单文件最大 5MB，自动轮转保留最近 7 天。

8.2 多显示器与混合 DPI

声明 Per-Monitor DPI Aware V2（manifest 或 SetProcessDpiAwarenessContext(DPI_AWARENESS_CONTEXT_PER_MONITOR_AWARE_V2)）。Overlay 创建时通过 GetDpiForWindow(hwndDialog) 获取对话框 DPI，按比例缩放所有尺寸。

8.3 安全软件兼容

可能触发启发式检测的行为：

  • 全局 SetWinEventHook 监控所有窗口事件
  • SendInput 模拟键盘输入
  • UI Automation 跨进程读取/写入控件

缓解措施：

  • 代码签名（EV 证书优先）
  • manifest 声明所需权限（uiAccess=false）
  • 只在检测到文件对话框时才进行 UI Automation 操作，不做全局扫描
  • 日志记录所有操作，方便用户向安全软件厂商申诉

9. 白名单 JSON Schema

白名单文件 whitelist.json，外置于 exe 旁，支持热更新（修改后无需重启，下次检测时重新加载）。

{
  "version": 1,
  "apps": [
    {
      "name": "Google Chrome",
      "process_name": "chrome.exe",
      "enabled": true,
      "verified": true,
      "detect": {
        "window_class": "Chrome_WidgetWin_1",
        "uia_conditions": [
          { "role": "Edit", "name_contains": "文件名|File name" },
          { "role": "Button", "name_contains": "打开|Open|保存|Save" }
        ],
        "min_match_count": 2
      },
      "navigate": {
        "method": "uia_set_value",
        "target_control": {
          "role": "Edit",
          "name_contains": "文件名|File name"
        },
        "confirm": {
          "role": "Button",
          "name_contains": "打开|Open|保存|Save"
        }
      }
    },
    {
      "name": "Visual Studio Code",
      "process_name": "Code.exe",
      "enabled": true,
      "verified": true,
      "detect": {
        "uia_conditions": [
          { "role": "Edit", "automation_id": "1148" },
          { "role": "Button", "name_contains": "打开|Open" }
        ],
        "min_match_count": 2
      },
      "navigate": {
        "method": "clipboard_set",
        "target_control": { "role": "Edit" },
        "confirm": { "method": "send_keys", "keys": "{ENTER}" }
      }
    },
    {
      "name": "Microsoft Word",
      "process_name": "WINWORD.EXE",
      "enabled": true,
      "verified": false,
      "detect": {
        "window_class": "NUIDialog",
        "uia_conditions": [
          { "role": "Edit", "name_contains": "文件名" },
          { "role": "Button", "name_contains": "打开" }
        ],
        "min_match_count": 2
      },
      "navigate": {
        "method": "uia_set_value",
        "target_control": { "role": "Edit", "name_contains": "文件名" },
        "confirm": { "role": "Button", "name_contains": "打开" }

}

9.1 字段说明

字段类型必填说明
apps[].namestring是应用显示名称，仅用于标识
apps[].process_namestring是进程名（含 .exe），用于快速匹配前台窗口
apps[].enabledbool否是否启用，默认 true
apps[].verifiedbool否是否经过实际测试，默认 false
apps[].detect.window_classstring否窗口类名精确匹配
apps[].detect.uia_conditionsarray是UI Automation 控件特征列表
apps[].detect.min_match_countint否最少命中数，默认 uia_conditions 长度
apps[].navigate.methodstring是"uia_set_value" 或 "clipboard_set"
apps[].navigate.target_controlobject是目标编辑框的 UIA 特征
apps[].navigate.confirmobject是确认操作：Button Invoke 或 send_keys

9.2 uia_conditions 语法

字段说明
roleUIA 控件类型：Edit, Button, ComboBox, TabItem 等
name_contains控件 Name 属性包含的文本，多语言用 | 分隔
automation_id控件 AutomationId 精确匹配
class_name控件 ClassName 精确匹配

10. 开发路线图

P0 — 基础可用 (2-3 周)
  • 标准对话框检测（#32770 + 控件特征验证）
  • IShellWindows 遍历获取资源管理器打开路径
  • 基础 Overlay 窗口（Win32 宿主 + 简单 ListView，先不用 WinUI 3）
  • 标准对话框导航注入（SendMessage WM_SETTEXT + BM_CLICK）
  • Named Mutex 防多开
  • 系统托盘图标 + 退出菜单
